# Releasing

CHord publishes to Maven Central through the Sonatype Central Publisher Portal under the
namespace `io.github.orhaugh`.

## One time setup

1. Sign in to [central.sonatype.com](https://central.sonatype.com) with the GitHub account that
   owns this repository; the `io.github.orhaugh` namespace is verified through GitHub.
2. Generate a publishing token (user token) in the portal.
3. Create a GPG key pair for release signing and publish the public key to
   `keys.openpgp.org` or `keyserver.ubuntu.com`.
4. Add repository secrets in GitHub (Settings, Secrets and variables, Actions):

| Secret | Content |
|---|---|
| `CENTRAL_TOKEN_USERNAME` | Central portal token username |
| `CENTRAL_TOKEN_PASSWORD` | Central portal token password |
| `GPG_PRIVATE_KEY` | ASCII armoured private key (`gpg --armor --export-secret-keys KEYID`) |
| `GPG_PASSPHRASE` | Passphrase of that key |

Release credentials never live in the repository.

## Version policy

- `0.x` while the API is under development; anything may change between 0.x releases.
- Release candidates precede 1.0.0.
- Semantic versioning from 1.0.0, with API compatibility enforced in the build.
- The 1.0.0 gate list (stable APIs, SELECT and INSERT in production quality, TLS, pooling,
  compression, retry classification, failover, full common type coverage, passing soak and fault
  injection suites, no known data corruption bug) is defined by the project plan and is not
  negotiable.

## Cutting a release

1. Ensure `main` is green, including the nightly compatibility workflow.
2. Update the version: `./mvnw versions:set -DnewVersion=<version> -DprocessAllModules
   -DgenerateBackupPoms=false` (and commit), move the `Unreleased` changelog section under
   the new version with a date.
3. Refresh `project.build.outputTimestamp` in the parent POM (reproducible builds).
4. Local validation without uploading:

   ```bash
   ./mvnw -Prelease verify
   ./mvnw -Prelease deploy -DskipPublishing=true
   ```

5. Tag signed: `git tag -s v<version> -m "CHord <version>"` and push the tag.
6. Run the Release workflow (Actions, Release) with `dry_run: true` first; inspect the uploaded
   artifact bundle (jars, sources, javadoc, signatures, SBOM).
7. Run it again with `dry_run: false`. The build uploads a bundle to the Central portal with
   `autoPublish` disabled; review the validated deployment in the portal and press publish.
8. Back on `main`: bump to the next `-SNAPSHOT`, restore the `Unreleased` changelog section.

## After the first release

- Enable API compatibility checking: set the japicmp plugin `skip` to `false` in the parent POM
  with the released version as the baseline, so incompatible changes fail the build.
- Attach the generated CycloneDX SBOMs and signed artifacts to the GitHub release created from
  the tag.

## What every published artifact contains

Main jar, sources jar, javadoc jar, POM with complete name, description, URL, licence, developer,
SCM and issue management metadata, PGP signatures for all of them, and checksums generated by the
portal upload. The published set is `chord-protocol`, `chord-transport`, `chord-codec`,
`chord-client`, `chord-jdbc`, `chord-observability`, `chord-testkit` and `chord-bom`; the
`chord-examples` and `chord-benchmarks` modules are never published. `chord-bom` lists exactly
the published set so consumers can align versions.

Local validation on a machine without the release GPG key: append `-Dgpg.skip=true` to the
commands above; signatures are then exercised only by the workflow's dry run.
